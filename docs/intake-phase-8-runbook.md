# Intake Phase 8 Runbook — Cold Steel II End-to-End

This runbook is the manual procedure for executing **PR 8** of the Walkthrough Intake System rollout: regenerating `walkthroughs/trails-of-cold-steel-ii/main-walkthrough.json` from the source markdown pages using the new deterministic pipeline, seeding `tools/intake/training-data.json` from the corrections made during review, and adding the end-to-end vitest spec that gates future regressions.

Phase 8 is gated on PRs 1–7 being merged to `main`. The agent that opens PR 8 can write the test file and fixtures, but the **review pass and webapp smoke test are human steps** — that's the point of phase 8.

## Prerequisites

```bash
# 1. Pull the merged main
git checkout main
git pull

# 2. Install both packages
cd tools/intake && npm install && cd -
cd tools/intake-extension && npm install && cd -

# 3. Verify everything still builds + tests pass
cd tools/intake && npx tsc --noEmit && npm test && cd -
cd tools/intake-extension && npm test && cd -
```

If any of those steps fail, **stop** and fix them before continuing.

> **Windows note:** PowerShell aliases `curl` to `Invoke-WebRequest`, which has different syntax. Everywhere this runbook says `curl`, use one of:
>
> - `curl.exe` (the real curl, ships with Windows 10+): `curl.exe -X POST http://localhost:3847/api/convert`
> - `Invoke-RestMethod`: `Invoke-RestMethod -Method Post -Uri http://localhost:3847/api/convert`
>
> Or run these steps in Git Bash / WSL where `curl` behaves as expected.

## Step 1 — Delete the old walkthrough files and start fresh

The old `main-walkthrough.json` was generated by the prior AI pipeline. Remove it (and any leftover `.intake/` state) so you're working from a clean slate:

```bash
rm -f walkthroughs/trails-of-cold-steel-ii/main-walkthrough.json
rm -rf walkthroughs/trails-of-cold-steel-ii/.intake/
```

PowerShell equivalent:

```powershell
Remove-Item walkthroughs/trails-of-cold-steel-ii/main-walkthrough.json -ErrorAction SilentlyContinue
Remove-Item walkthroughs/trails-of-cold-steel-ii/.intake -Recurse -Force -ErrorAction SilentlyContinue
```

After this, the only files remaining under `walkthroughs/trails-of-cold-steel-ii/` should be the source page markdown files (`page1.md` … `page12.md`).

## Step 2 — Start the intake server

From the repo root:

```bash
npx tsx tools/intake/src/cli.ts start \
  --game "Trails of Cold Steel II" \
  --source "https://www.example.com/trails-of-cold-steel-ii-walkthrough"
```

Replace the `--source` URL with the actual upstream source — it ends up in the attribution string of the generated JSON, so use the real URL the pages came from.

The server boots on `http://localhost:3847` and creates `walkthroughs/trails-of-cold-steel-ii/.intake/{pages,session.json}`. Leave this terminal running.

## Step 3 — Seed the captured pages

The 12 Cold Steel II source pages already live at `walkthroughs/trails-of-cold-steel-ii/page1.md` … `page12.md`. We don't need the browser extension for this rerun — the markdown is already on disk. POST each page directly:

In a **second** terminal:

```bash
for i in $(seq 1 12); do
  jq -Rs --arg n "$i" --arg t "Cold Steel II — Page $i" \
    '{ title: $t, url: ("file://page" + $n + ".md"), markdown: . }' \
    walkthroughs/trails-of-cold-steel-ii/page$i.md \
  | curl -s -X POST http://localhost:3847/api/intake \
      -H 'Content-Type: application/json' -d @-
  echo
done

# Sanity check — should report 12 pages
curl -s http://localhost:3847/api/pages | jq 'length'
```

PowerShell equivalent (if you're on Windows):

```powershell
for ($i=1; $i -le 12; $i++) {
  $md = Get-Content "walkthroughs/trails-of-cold-steel-ii/page$i.md" -Raw
  $body = @{ title = "Cold Steel II — Page $i"; url = "file://page$i.md"; markdown = $md } | ConvertTo-Json
  Invoke-RestMethod -Method Post -Uri http://localhost:3847/api/intake -ContentType 'application/json' -Body $body
}
(Invoke-RestMethod http://localhost:3847/api/pages).Length
```

> If you'd rather practice the live workflow, follow `tools/intake-extension/README.md` instead and capture each page through the browser extension. The result is identical.

## Step 4 — Run the converter

```bash
curl -X POST http://localhost:3847/api/convert
```

PowerShell: `curl.exe -X POST http://localhost:3847/api/convert` or `Invoke-RestMethod -Method Post -Uri http://localhost:3847/api/convert`

Example output:

```json
{ "success": true, "sections": 1, "total_blocks": 809 }
```

**Understanding the section count:** The converter joins all captured pages into a single document and then runs `detect-sections.ts`, which splits on `##` (H2) heading boundaries. In practice the converter may not produce one section per page — it depends on how the combined document's heading structure is parsed. A single section with hundreds of blocks is a valid (and common) result for a first run.

### Verifying pages were captured

Before investigating section counts, confirm all 12 pages made it into the server:

```bash
curl -s http://localhost:3847/api/pages | jq 'length'
```

PowerShell: `(Invoke-RestMethod http://localhost:3847/api/pages).Count`

If the count is less than 12, re-seed the missing pages (Step 3).

### Inspecting the output

```bash
curl -s http://localhost:3847/api/sections | jq '.[] | {id, title, blocks: (.blocks|length)}'
```

If you get fewer sections than expected:

1. **Check the source headings** — verify the page markdown actually has `## ` (H2) lines. Some capture methods (Readability/Turndown) may strip or downgrade headings.
   ```bash
   curl -s http://localhost:3847/api/pages/1 | jq -r '.markdown' | Select-String "^## "
   ```

2. **Section count ≠ page count** — the converter treats all pages as one document. A walkthrough with one long narrative and few H2 breaks will naturally produce fewer sections.

3. **This is OK to proceed with** — even 1 section with 809 blocks is a valid starting point for the review workflow in Step 5. You can still approve and finalize.

## Step 5 — Review and correct sections

This is the human-judgment step. Walk through each section and verify the classifier got it right. Use whatever workflow you prefer — here's a curl-driven loop:

```bash
# Dump every section as one big JSON to skim
curl -s http://localhost:3847/api/sections > /tmp/cs2-sections.json
$EDITOR /tmp/cs2-sections.json

# For each block that's wrong, PUT a correction:
curl -X PUT http://localhost:3847/api/sections/<section-id>/blocks/<index> \
  -H 'Content-Type: application/json' \
  -d '{
    "block": { "type": "encounter", "name": "Magic Knight Ortheim", "stats": {"HP":"12000","Weakness":"Earth"} },
    "approved": true
  }'

# Once every block in a section looks right, approve the whole thing:
curl -X POST http://localhost:3847/api/approve/<section-id>
```

What to look for:

| Block type | Heuristic |
|---|---|
| **prose** | Default — narrative direction ("Head north to…") |
| **encounter** | Boss/miniboss fights with stats. Should have `name` and `stats` populated. |
| **quest** | Side quests. Check `quest_type` — `missable` for chapter-locked, `side` for evergreen. |
| **event** | Cold Steel bonding events especially — verify `event_type: 'bonding'`, `missable: true`, and `availability` describes the window. |
| **table** | Shop inventories, item lookups. Should NOT be an encounter. |
| **checklist** | Treasure chests, NPC visits, collectibles — anything you'd want a checkbox for. |
| **callout** | Warnings/tips/notes — usually short, often started with "Warning:" or "Note:". |

When you correct a block via `PUT`, the proposal envisions that correction being recorded as a `TrainingExample` so the rules engine can learn. The current server's PUT endpoint only updates `sections.json` — it does **not** yet write to `training-data.json` automatically. To seed training corrections for this PR, either:

- Add them by hand to `tools/intake/training-data.json` using the `TrainingExample` shape (see `src/types.ts`).
- Or write a small one-off Node script that diffs the original vs. corrected `sections.json` and emits the examples.

Decide which corrections are worth keeping (high-confidence patterns that will recur), and commit those.

## Step 6 — Set the graduation threshold (optional)

If this is your first real run and you want the converter to graduate quickly on future walkthroughs:

```bash
npx tsx tools/intake/src/cli.ts set-threshold 5
npx tsx tools/intake/src/cli.ts training-status
```

For a serious training run, 50 or 100 is the common choice.

## Step 7 — Finalize

```bash
curl -X POST http://localhost:3847/api/finalize
```

This writes `walkthroughs/trails-of-cold-steel-ii/main-walkthrough.json`. The session state flips to `finalized`. You can now stop the server (Ctrl+C in the first terminal).

## Step 8 — Validate the generated walkthrough

Since you deleted the old file in Step 1, there's nothing to diff against — validate the fresh output on its own merits:

```bash
# Quick structural sanity check — section count and block totals
jq '{ sections: (.sections | length), blocks: [.sections[].blocks | length] | add }' \
  walkthroughs/trails-of-cold-steel-ii/main-walkthrough.json

# Validate against the project schema
npx ajv validate -s walkthroughs/walkthrough.schema.json \
  -d walkthroughs/trails-of-cold-steel-ii/main-walkthrough.json
```

Verify:

- No AI hallucinations (every block traces back to source page content)
- No dropped sections, encounters, or quests
- Block types match the patterns from Step 5
- Section IDs are stable across runs (rerun convert+finalize, ensure same IDs)
- At least one encounter block present per major act/chapter

> **Tip:** If you kept a backup of the old file elsewhere (e.g. in a prior git commit), you can still diff with `git show main:walkthroughs/trails-of-cold-steel-ii/main-walkthrough.json > /tmp/cs2-old.json` and compare.

## Common pitfalls

| Pitfall | Mitigation |
|---|---|
| Running `set-threshold` from inside `tools/intake/` | Run from repo root — the CLI resolves `training-data.json` relative to cwd |
| Port 3847 already in use | Pass `--port <n>` to `start`; update the extension if you use it too |
| Forgot to capture a page | `POST /api/intake` with the missing page — the page_number auto-increments |
| Reviewed sections but didn't approve them | `/api/finalize` doesn't filter by `approved` flag — it includes all sections. The `approved` flag is for tracking, not gating. Decide policy before relying on it. |
| Old `.intake/` directory from a previous run | Delete `walkthroughs/trails-of-cold-steel-ii/.intake/` before `start` to begin fresh |
| `main-walkthrough.json` validates but webapp can't render | Compare a new block's shape against an old one — schema validation doesn't catch every contract issue |
